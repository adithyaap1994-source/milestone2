# Deployment Plan: Mutual Fund FAQ Assistant

This plan defines production deployment for:
- **Scheduler**: GitHub Actions
- **Backend API**: Render
- **Frontend**: Vercel

## 1) Target Deployment Architecture

```text
GitHub Actions (scheduled ingestion)
    -> runs Phase 1 pipeline
    -> writes ingestion artifacts/reports

Vercel (Next.js frontend)
    -> calls Render backend API over HTTPS

Render (Python backend API)
    -> serves chat/session APIs
    -> reads local artifacts generated by ingestion flow
```

## 2) Environments

Use three environments:
- **dev**: frequent testing, lower thresholds
- **staging**: production-like validation
- **prod**: live deployment with strict release gates

Recommended branches:
- `develop` -> dev
- `staging` -> staging
- `main` -> prod

## 3) Scheduler Deployment (GitHub Actions)

### Workflow Setup

- Keep scheduler workflow in `.github/workflows/daily-ingestion.yml`
- Trigger:
  - `schedule` (daily run at required time)
  - `workflow_dispatch` (manual rerun/backfill)
- Use `concurrency` group to avoid overlapping runs.

### Scheduler Responsibilities

- Install dependencies
- Run Phase 1 ingestion pipeline
- Generate artifacts:
  - normalized docs
  - chunk payloads
  - index payloads
  - ingestion report
- Upload reports as workflow artifacts

### Reliability Controls

- Retries for transient network fetch failures
- Fast failure on pipeline exceptions
- Store full logs per run
- Optional notifications on failure (GitHub email/Slack webhook)

## 4) Backend Deployment (Render)

### Service Type

- Deploy as a **Web Service** on Render.
- Runtime: Python 3.11+ recommended.
- Entry command should start Phase 2 API service.

### Build and Start

- Build command:
  - `pip install -r phase-2/requirements.txt`
- Start command:
  - `python phase-2/run_phase2_ui.py`
- The API listens on **`0.0.0.0`** and uses the **`PORT`** environment variable (set by Render). Local default remains `8080` if `PORT` is unset.

### Infrastructure as Code (`deploy/render.yaml`)

The **Render Blueprint** lives at **`deploy/render.yaml`** (not the repo root) so **Vercel** imports that use the repository root do not pick up a root `render.yaml` and try to run the **Python** builder (which fails with “No python entrypoint found”). Render still supports blueprints from a subpath when you specify it in the Blueprint flow.

**Create or update the service from the dashboard**

1. Sign in at [Render Dashboard](https://dashboard.render.com).
2. **New +** → **Blueprint** (or **Web Service** if you prefer manual setup).
3. Connect the GitHub repository, select branch **`main`**, and set the blueprint file path to **`deploy/render.yaml`** when prompted (or paste its contents).
4. Apply the blueprint; Render will create **`mutual-fund-faq-api`** (you can rename the service in the UI).
5. After the first deploy succeeds, copy the public URL (e.g. `https://mutual-fund-faq-api.onrender.com`) for Vercel: set `NEXT_PUBLIC_API_BASE_URL` to that origin (no trailing slash).

**Manual Web Service** (same as blueprint, if not using Blueprint)

- **Root directory**: repository root (default).
- **Build command**: `pip install -r phase-2/requirements.txt`
- **Start command**: `python phase-2/run_phase2_ui.py`
- **Health check path**: `/health`

### Index data on the server

Retrieval reads `phase-1/data/index/keyword_payload.jsonl` relative to the repo root. Those JSONL files are intentionally gitignored until you choose a distribution strategy. Until they exist on the instance, the API still runs and `/health` succeeds; factual answers may return **insufficient evidence** until you either commit a small approved index for staging, restore artifacts to disk during deploy, or sync from object storage in a follow-up step.

### Required Environment Variables

Define on Render:
- `PYTHONUNBUFFERED=1`
- Any additional app config variables you introduce later

### Health and Routing

- Use health endpoint:
  - `/health`
- Public API endpoints:
  - `POST /api/thread`
  - `POST /api/chat`
  - `GET /api/thread/{thread_id}`

### Backend Production Notes

- Use persistent disk or external storage if long-term thread/audit retention is needed.
- If using ephemeral filesystem only, treat thread/audit files as non-durable runtime state.
- Add request timeout and basic rate limiting at platform/app layer.

## 5) Frontend Deployment (Vercel)

### Project Setup

- **Root directory** (Vercel project setting): **`phase-2/frontend`** (recommended). The app’s own `phase-2/frontend/vercel.json` then applies.
- **If you leave Root Directory at the repository root** (common mistake): the repo has a **root `package.json`** with an **npm workspace** for `phase-2/frontend` plus a **root `vercel.json`** with **`framework: "nextjs"`** and default **`npm install` / `npm run build`**. Together they force a **Node/Next** build so Vercel does not treat tracked `phase-*/requirements.txt` files as a Python app (which causes “No python entrypoint found”).
- **Framework preset**: Next.js (auto-detected from `package.json` / `vercel.json`).
- The Render blueprint is **`deploy/render.yaml`**, not `render.yaml` at the root, so Vercel does not mis-detect the monorepo as a Python Render service.

### Connect the repository (dashboard)

1. Sign in at [Vercel](https://vercel.com) → **Add New…** → **Project**.
2. **Import** the GitHub repository that contains this monorepo.
3. Under **Configure Project**:
   - Set **Root Directory** to **`phase-2/frontend`** (Edit → select folder). Vercel must run commands from this directory so `next build` resolves `app/` and dependencies correctly.
   - **Framework Preset**: Next.js (should auto-detect).
   - **Build Command**: `npm run build` (default).
   - **Install Command**: `npm install` (default).
4. **Environment Variables** (before first deploy):
   - Name: `NEXT_PUBLIC_API_BASE_URL`
   - Value: your Render backend origin, e.g. `https://mutual-fund-faq-api.onrender.com` (no trailing slash; use **https**).
   - Add for **Production** and **Preview** (and **Development** if you use `vercel dev`), so preview deployments can call a staging API if you configure one.
5. **Deploy**. After the build, open the `.vercel.app` URL and verify the UI loads and can create a thread (browser calls the Render API; CORS on the Phase 2 server must allow the Vercel origin — current backend uses `Access-Control-Allow-Origin: *`).

### Environment Variables (Vercel)

- `NEXT_PUBLIC_API_BASE_URL=https://<render-backend-domain>`

See `phase-2/frontend/.env.example` for a template. Values prefixed with `NEXT_PUBLIC_` are exposed to the browser; do not put secrets in them.

### CLI (optional)

From your machine, with [Vercel CLI](https://vercel.com/docs/cli) installed:

```bash
cd phase-2/frontend
vercel link   # once per project
vercel --prod
```

Set `NEXT_PUBLIC_API_BASE_URL` in the linked project on Vercel or via `vercel env add`.

### Build

- Install command: `npm install`
- Build command: `npm run build`
- Output: Next.js default

### Frontend Production Notes

- Verify CORS on backend allows Vercel origin.
- Add environment variables separately per Vercel environment (Preview/Production).
- Point Preview `NEXT_PUBLIC_API_BASE_URL` at a staging Render service if you split environments; Production should use the production API URL.

## 6) CI/CD Flow

### On Backend Changes

1. Merge to target branch
2. Render auto-deploys backend
3. Run smoke tests against deployed backend:
   - create thread
   - send factual query
   - send advisory query
   - load thread history

### On Frontend Changes

1. Merge to target branch
2. Vercel builds Preview/Production deployment
3. Validate UI:
   - session creation
   - query send
   - thread switching

### On Ingestion/Scheduler Changes

1. Merge workflow/pipeline updates
2. Trigger manual `workflow_dispatch` run
3. Validate artifacts and logs
4. Allow next scheduled run

## 7) Release Gate Checklist

Before promoting to production:

- Backend `/health` returns success
- Frontend can call backend successfully
- No advisory leakage in policy test prompts
- Factual responses satisfy citation + last_updated contract
- Multi-thread isolation verified
- Latest ingestion run completed successfully
- Phase 3 release metrics reviewed and approved

## 8) Rollback Plan

### Backend Rollback (Render)

- Redeploy previous stable Render deployment from dashboard.
- Confirm `/health` and key API endpoints.

### Frontend Rollback (Vercel)

- Promote previous successful deployment from Vercel dashboard.
- Re-verify API connectivity.

### Scheduler Rollback

- Revert workflow/pipeline commit in GitHub
- Trigger manual ingestion run to verify restored behavior

## 9) Monitoring and Ops

Track:
- Scheduler run success/failure rate
- Ingestion artifact freshness
- Backend API uptime and error rates
- Frontend availability
- Policy incidents (advisory leakage)
- Citation validity rate

Minimum alerting:
- Scheduler failure
- Backend health endpoint failure
- Spike in API 5xx errors
- Drop in citation compliance metrics

## 10) Implementation Sequence

1. Deploy backend to Render (staging)
2. Deploy frontend to Vercel (staging) and connect to staging backend
3. Configure and validate GitHub Actions scheduler
4. Run end-to-end smoke suite
5. Promote backend and frontend to production
6. Enable daily scheduler in production branch
