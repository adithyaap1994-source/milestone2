# Deployment Plan: Mutual Fund FAQ Assistant

This plan defines production deployment for:
- **Scheduler**: GitHub Actions
- **Backend API**: Render
- **Frontend**: Vercel

## 1) Target Deployment Architecture

```text
GitHub Actions (scheduled ingestion)
    -> runs Phase 1 pipeline
    -> writes ingestion artifacts/reports

Vercel (Next.js frontend)
    -> calls Render backend API over HTTPS

Render (Python backend API)
    -> serves chat/session APIs
    -> reads local artifacts generated by ingestion flow
```

## 2) Environments

Use three environments:
- **dev**: frequent testing, lower thresholds
- **staging**: production-like validation
- **prod**: live deployment with strict release gates

Recommended branches:
- `develop` -> dev
- `staging` -> staging
- `main` -> prod

## 3) Scheduler Deployment (GitHub Actions)

### Workflow Setup

- Keep scheduler workflow in `.github/workflows/daily-ingestion.yml`
- Trigger:
  - `schedule` (daily run at required time)
  - `workflow_dispatch` (manual rerun/backfill)
- Use `concurrency` group to avoid overlapping runs.

### Scheduler Responsibilities

- Install dependencies
- Run Phase 1 ingestion pipeline
- Generate artifacts:
  - normalized docs
  - chunk payloads
  - index payloads
  - ingestion report
- Upload reports as workflow artifacts

### Reliability Controls

- Retries for transient network fetch failures
- Fast failure on pipeline exceptions
- Store full logs per run
- Optional notifications on failure (GitHub email/Slack webhook)

## 4) Backend Deployment (Render)

### Service Type

- Deploy as a **Web Service** on Render.
- Runtime: Python 3.11+ recommended.
- Entry command should start Phase 2 API service.

### Build and Start

- Build command:
  - `pip install -r phase-2/requirements.txt`
- Start command:
  - `python phase-2/run_phase2_ui.py`

### Required Environment Variables

Define on Render:
- `PYTHONUNBUFFERED=1`
- Any additional app config variables you introduce later

### Health and Routing

- Use health endpoint:
  - `/health`
- Public API endpoints:
  - `POST /api/thread`
  - `POST /api/chat`
  - `GET /api/thread/{thread_id}`

### Backend Production Notes

- Use persistent disk or external storage if long-term thread/audit retention is needed.
- If using ephemeral filesystem only, treat thread/audit files as non-durable runtime state.
- Add request timeout and basic rate limiting at platform/app layer.

## 5) Frontend Deployment (Vercel)

### Project Setup

- Root directory: `phase-2/frontend`
- Framework preset: Next.js

### Environment Variables (Vercel)

- `NEXT_PUBLIC_API_BASE_URL=https://<render-backend-domain>`

### Build

- Install command: `npm install`
- Build command: `npm run build`
- Output: Next.js default

### Frontend Production Notes

- Verify CORS on backend allows Vercel origin.
- Add environment variables separately per Vercel environment (Preview/Production).

## 6) CI/CD Flow

### On Backend Changes

1. Merge to target branch
2. Render auto-deploys backend
3. Run smoke tests against deployed backend:
   - create thread
   - send factual query
   - send advisory query
   - load thread history

### On Frontend Changes

1. Merge to target branch
2. Vercel builds Preview/Production deployment
3. Validate UI:
   - session creation
   - query send
   - thread switching

### On Ingestion/Scheduler Changes

1. Merge workflow/pipeline updates
2. Trigger manual `workflow_dispatch` run
3. Validate artifacts and logs
4. Allow next scheduled run

## 7) Release Gate Checklist

Before promoting to production:

- Backend `/health` returns success
- Frontend can call backend successfully
- No advisory leakage in policy test prompts
- Factual responses satisfy citation + last_updated contract
- Multi-thread isolation verified
- Latest ingestion run completed successfully
- Phase 3 release metrics reviewed and approved

## 8) Rollback Plan

### Backend Rollback (Render)

- Redeploy previous stable Render deployment from dashboard.
- Confirm `/health` and key API endpoints.

### Frontend Rollback (Vercel)

- Promote previous successful deployment from Vercel dashboard.
- Re-verify API connectivity.

### Scheduler Rollback

- Revert workflow/pipeline commit in GitHub
- Trigger manual ingestion run to verify restored behavior

## 9) Monitoring and Ops

Track:
- Scheduler run success/failure rate
- Ingestion artifact freshness
- Backend API uptime and error rates
- Frontend availability
- Policy incidents (advisory leakage)
- Citation validity rate

Minimum alerting:
- Scheduler failure
- Backend health endpoint failure
- Spike in API 5xx errors
- Drop in citation compliance metrics

## 10) Implementation Sequence

1. Deploy backend to Render (staging)
2. Deploy frontend to Vercel (staging) and connect to staging backend
3. Configure and validate GitHub Actions scheduler
4. Run end-to-end smoke suite
5. Promote backend and frontend to production
6. Enable daily scheduler in production branch
